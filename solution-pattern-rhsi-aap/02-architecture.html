Using Red Hat Service Interconnect to extend the reach of Ansible Automation Platform

Architecture

Introduction for the architecture of this solution pattern.

1. Technology Stack

  • Red Hat supported products

    • Red Hat Service Interconnect

    • Red Hat Ansible Automation Platform

    • Red Hat OpenShift

    • Red Hat Enterprise Linux

  • Other open source products:

    • Skupper

    • Podman

2. An in-depth look at the solution’s architecture

The trusted partner has a Red Hat OpenShift cluster running on the public cloud. In this Red Hat OpenShift cluster, they have a Red Hat Ansible Automation Platform (AAP) instance used to maintain third-party infrastructures.

This AAP instance runs inside the aap namespace. For their AAP instance to be able to communicate with third-party servers, that are not accessible from the public cloud, they will use Red Hat Service Interconnect (RHSI) version 2.

A separate namespace will be created for each isolated third-party network.

Network policies will be applied to the third-party namespaces, to ensure that incoming internal traffic will only be allowed from the aap namespace. This will ensure that unauthorized access from a separate third-party namespace, as well as from any other namespace within this OpenShift cluster, will be denied.

Each third-party namespace will have an instance of Red Hat Service Interconnect (RHSI) and this instance will be responsible for receiving an incoming RHSI link, coming from the third-party network. Once the Red Hat Service Interconnect link is established, using mutual TLS to ensure privacy and security, a selected set of servers, running in the third-party network, will be exposed by an RHSI instance running within the third-party network, into the third-party specific namespace on the Red Hat OpenShift cluster.

The respective servers are then exposed as Kubernetes services within the third-party namespace, allowing the AAP instance to communicate with them via SSH transparently through RHSI.

Distinct inventories will be created inside AAP, for each third-party network, representing the servers to be managed.

2.1. Alternative architecture

Consider an alternative and illustrative architecture, for scenarios where the Red Hat Ansible Automation Platform instance, used by the Trusted Partner, is not hosted in a Public Cloud environment.

By introducing an intermediary site within a Public Cloud, we can seamlessly integrate private networks. This approach involves deploying another Red Hat OpenShift cluster in the Public Cloud, with dedicated namespaces for each third-party network. These namespaces have their own Red Hat Service Interconnect instance to provide secure and isolated endpoints for incoming links from both the Trusted Partner network and the third-party networks.

While this alternative approach is not the focus of this solution pattern, it highlights the flexibility and adaptability of our solution to diverse network environments. By leveraging cloud-based infrastructure, we can empower organizations to achieve optimal performance, security, and scalability.

3. About the Technology Stack

Red Hat Service Interconnect provides access to selected workloads running in the third-party network.
In this solution pattern, we illustrate two third-party networks being managed by AAP. One of them, represented as Third-party network A has an RHSI instance running on a Red Hat Enterprise Linux (RHEL) machine.

The following one, represented as Third-party network B has an RHSI instance running using Podman.

It is important to mention that Red Hat Service Interconnect (version 2+) runs on any Kubernetes cluster (like Red Hat OpenShift), Red Hat Enterprise Linux and using Podman or Docker container engines.
